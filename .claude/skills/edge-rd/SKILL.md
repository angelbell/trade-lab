# edge-rd — 研究工程表（CLAUDE.md の規律を工程として強制する）

引数でモードを選ぶ: `/edge-rd idea <テーマ>` = モードA、`/edge-rd test <仮説>` = モードB。
引数なし・不明瞭なら、どちらのモードかをまず1問で確認する。

## 共通の鉄則（両モード）
1. 数字の出所は「いま実行したスクリプトの出力」のみ。記憶・推定の数字は必ず「未検証」と明記する。
2. 案・仮説を出す前に `docs/verified_findings.md` を grep し、既検証のもの・既に死んだ親戚を候補ごとに付記する（再検証はユーザーが最も嫌うコスト）。
3. 1メッセージ1実験。複数の判断を連鎖させない。adopt / kill / close の判断は毎回ユーザーが行う。自分から「決着」を宣言しない。
4. 報告は毎回: 何をしたのか（手順）＋ロジック（売買ルールを平文で: 何が起きたら・どこで入り・どこで損切り・どこで利確・どのゲート）＋ PF・N（N/年）・リスク（DD or 賭け率）をTF別に。分布は中央値・標準偏差併記。地の文は日本語、英語ジャーゴン混ぜない、番号は 1. 2. 3.。
5. 結果が悪くても手を止めない: 「なぜ落ちたか＝病巣」を特定し、それを直接叩く次実験を提案する。ただし次を撃つ前に仕様確認。

## モードA — アイデア発散（新しい案・組み合わせ）
1. 軸ごとに発散する（収束を先にしない）: いつ入るか（レジーム/時間帯/イベント近接）・何を取引するか（銘柄/合成/サイド）・出口・執行（指値/分割/再武装）・組み合わせ、の各軸で2〜3案、合計10案程度。
2. 各案は必ずこの4行で書く（docs/proposals.md と同じ形式）:
   - 機構: なぜ効くはずか（因果の一文）
   - 残存理由: 世界中で採掘済みという事前確率に対し、なぜ残っていそうか（容量小/執行壁/若い市場/裁量にしか見えなかった等）
   - 最安の殺し方: 30秒〜1スクリプトで殺せる最初のスクリーン（mfe_mae.py / edge_harness.py 優先）
   - 予想される死に方: 先に予言を書く（外れたら学習が残る）
3. エッジの大きさ×独立性（既存ブックとの相関予測）×検証コストで自己採点し、上位2案だけ深掘り提案。1トレードあたりの大きさが薄い案（multi-R未満のintraday）は下位に落とす。
4. 採用された案は docs/proposals.md に追記してからモードBへ。

## モードB — 実験（設計→承認→実行→検証→記録）
1. **仕様カードを先に出し、ユーザーのOKまで実装しない。** カードの必須項目:
   - 対象セル（銘柄×TF×サイド）と全期間・全シグナルのベース
   - 固定するもの（entry/stop/コスト）と動かすもの（1実験1レバー）
   - 事前登録の合格線と死に方（PASS/KILL条件を数字で凍結。走った後に緩めない）
   - 流用するスクリプト（車輪の再発明禁止: breakout_wave.py / ema_pullback.py / edge_harness.py / regime_discriminator.py / overfit_audit.py）
   - 予想結果
2. 実装・計測は原則 `measure` サブエージェント（.claude/agents/measure.md, sonnet）に投げる: 仕様カードを凍結して渡す。契約（既存関数のimport流用・スモーク→本番・docs編集禁止・判断語禁止）はエージェント定義側に書いてある。
3. **返ってきた数字は必ずローカル再実行で照合してから**台帳・報告に書く。
4. 反証チェックリスト（CLAUDE.md の 1〜11）を項目ごとに PASS / FAIL / 未実施 で列挙する。特に: 全シグナルベース→勝率vs損益分岐(1/(1+RR))→IS/OOS→±1プラトー→年別/時代別→コストは判定の後段で分離（「エッジ無し」と「エッジ有・コスト死」は別ラベル）→random-drop null は meanR でなく totR/DD で。
5. 判定が出たら: docs/verified_findings.md に1エントリ追記（tested→verdict形式、病巣と次のレバー候補まで）。proposals.md の該当項目の status を更新。kill でも病巣から次実験を1つ提案して終える。
