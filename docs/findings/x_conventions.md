### cost-per-stop-identity

- 🔬🔑🚨 **【病名の付け間違いの疑い】「低TFはコストで死ぬ」は、正しくは「損切りが小さいとコストで死ぬ」ではないか**（`scratchpad/gap_curve_screen.py`, 2026-07-13。ユーザーの「スキャルで50〜100pips狙う」という要求から逆算して発覚）。
  **恒等式**: `R = (値幅 − コスト) / 損切り幅` ∴ **コストの R 換算 = コスト ÷ 損切り幅**。**TF は式に入っていない。**
  **実証**: 入口9家族 × gold/BTC/USDJPY × 5分/15分 × long/short を、損切り `k×ATR(シグナルTF)`（k=1,2）で回したら **全216セルで PF(実コスト) < 1.13、PF素の中央値は 0.94〜0.98**＝コストが PF を **0.14〜0.44** 削っていた（BTC 5分: 素 0.97 → 実 **0.54**）。原因は **5分足の ATR が小さすぎること**: USDJPY 5分 ATR≒2.5pips → 損切り2〜5pips に対し往復0.9pip ＝ **コストが損切りの 18〜36%**。gold 5分 ATR≒$1.0 → $0.30 が **15〜30%**。
  **∴ 同じ5分足でも、損切りを 50〜100pips（ユーザーの要求スケール）に置けばコストは 1〜2% で無視できる。** 「低TF＝不利」ではなく「ATRベースの損切り＝低TFで小さくなる」だっただけの可能性。
  🚨 **要・再点検**: 台帳の「低TFはコストで死ぬ」系の判定（例: `gold200MAバウンスの5分化＝PF1.04`, s09）は、**すべて小さい損切りで測られている**。損切りを大きく取れば復活するものがあるかもしれない。**再テスト前にこの項を見ること。**

### r-unit-pullback-inflation

🚨🔑 **【規約】押し目指値の深さを R 単位で比べてはならない。** `walk()` の指値は
`lim = e − pf×(e − stop)` なので、**実現損切り幅は (1−pf) 倍に縮む** → R は 1/(1−pf) 倍に機械的に膨らむ。
実測（BTC h1 拡大足ショート k2.0 RR3）: **損切り幅の中央値 497.2（pf=0.25) → 139.4（pf=0.786）＝3.6倍の縮小**。
- 症状: R で測ると「押し目は深いほど良い」に見え、最良が pf=0.786 に出る。**同じトレードを入口価格に対する%
  で測り直すと最良は pf=0.382〜0.5 に移る**（0.786 は平均 +0.078% で明確に劣る）。
- さらに悪いのは帰無の側: 「成行（フル損切り幅）母集団からのランダム間引き」と R で比べると、
  **物差しの違う2つを比較している**（pf=0.786 なら 4.7倍）。当然 %ile 100 が出る。
- ∴ **ロット固定運用の評価は価格%（`pnl_px / e_real`）で行う。** サイズを risk に比例させる運用でのみ R は正しい。
- 兄弟＝法則7.5「サイズ倍率で割った量はサイズ規則を暗黙に含む」と同じ根。**単位はサイズ規則である。**

### mirror-cost-overcharge

🚨🔑 **【バグ・要注意】`mirror.invert()` の反転フレームで `walk()` のコストが過大請求になる。**
`walk()` は `R -= args.cost / risk * e_px` と計算するが、反転フレームの `e_px` は鏡像価格
（C = 2×全期間最高値 なので**実価格の約3倍**）。`mirror.py` の「比率ベースの特徴量は綺麗には鏡像化しない」の実例。
- 影響の大きさ: BTC h1 拡大足ショートで**符号が反転した**（過大請求版 PF 0.40〜0.64 の全滅 → 修正後 1.05〜1.28）。
  ショートを鏡像で回す実験は**すべてこの穴を通る**。
- 対処: **`walk` は `cost=0.0` で回し、外側で `pnl_px = R*risk − cost*e_real`（`e_real = C − e_px`）を引く。**
  `risk` は差分なので変換不要（無傷）。
- 検出法: ショートだけが不自然に全滅していたら、まずここを疑う。
